# AI-редактор для статей, новостей и вайбкодинга

## Описание проекта

Этот проект - локальный Python-агент для ежедневной работы с контентом: он собирает новости и статьи про ИИ, разработку и личные проекты в вайбкодинге, ранжирует темы, помогает выбрать самые востребованные инфоповоды, пишет черновики в авторском стиле, делает рерайт под разные площадки, добавляет картинки и ведёт журнал публикаций.

Основная идея: не просто генерировать посты, а собрать редакционную систему вокруг автора. Агент должен помогать находить хорошие темы, сравнивать их между собой, подстраивать текст под Telegram, VK, VC и Дзен, сохранять историю публикаций и постепенно учиться на правилах стиля автора.

## Как пользоваться

### 1. Добавить источники

Поддерживаемые типы источников:

- `rss` - обычные RSS-ленты.
- `website` - страница сайта, из которой агент собирает внутренние ссылки.
- `telegram` - публичный Telegram-канал через `https://t.me/s/...`.
- `apify_actor` - внешний Apify Actor из API Mega List.

Для Apify:

1. Создай аккаунт на `https://apify.com/`.
2. Возьми API token в Apify Console → Settings → API & Integrations.
3. Вставь token в `/admin/settings`, блок `Apify: внешние API-источники`.
4. На странице `/admin/sources` добавь пресеты:
   - `AI Hype Tracker`;
   - `Google News Scraper`;
   - `Hacker News Intelligence`.
5. Нажми `Проверить actor` в настройках перед регулярным запуском.

Важно: многие Apify Actors платные или pay-per-result. Для контроля расходов в настройках есть `Max items` и `Timeout`, а также выключатель `Apify sources`.

Для OSINT:

1. Открой `/admin/osint`.
2. Нажми `Обновить OSINT`.
3. Используй поиск по категориям: `Fact Checking`, `News`, `Telegram`, `GitHub`, `Image Search`, `Threat Intelligence`, `Domain and IP Research`.
4. Из найденного инструмента можно сделать черновик для блога, Telegram или Wiki.

Источник данных: публичный каталог `https://github.com/jivoi/awesome-osint`, который также использует `https://osint.juanmathewsrebellosantos.com/`. Применение в проекте ограничено легальным фактчекингом, проверкой источников, поиском публичных сигналов, defensive research и идеями для статей. Не использовать для доксинга, обхода приватности и несанкционированного доступа.

## Модель блога-лаборатории

Блог строится вокруг связки:

```text
Статья -> проект -> 3-5 проб -> заявка / доступ / расширенная версия
```

Как это работает:

- статья объясняет идею, проблему, подход и выводы;
- проект даёт человеку живой опыт, а не только описание;
- посетитель получает cookie `visitor_id`;
- для каждого проекта считается количество запусков в таблице `project_usage`;
- после лимита показывается экран окончания демо.

Практическая модель монетизации:

- `3` пробы для дорогих или ресурсоёмких AI-инструментов;
- `5` проб для простых демо и лид-магнитов;
- после лимита перевод в Telegram, заявку, донат, подписку или ручной доступ.

### Текст и рерайт

Рекомендуемый вариант сейчас - OpenRouter.

Базовые значения:

```text
Провайдер: OpenRouter
Base URL: https://openrouter.ai/api/v1
Model: openrouter/auto
```

Почему это удобно:

- можно быстро менять модели без изменения кода;
- можно использовать сильную модель для VC и Дзена;
- можно использовать более дешёвую или быструю модель для Telegram;
- можно тестировать разные модели под стиль автора.

Также поддерживаются:

- OpenAI;
- любой OpenAI-compatible endpoint, например локальный сервер или внешний шлюз.

### Картинки

Доступные варианты:

- `Fallback-обложка без API` - локальная генерация простой обложки;
- `OpenRouter Images` - качественная генерация через OpenRouter; текущий рабочий режим для красивых обложек: `recraft/recraft-v3`, 16:9, 1K;
- `Cloudflare Worker Images` - бесплатный быстрый fallback по схеме `saurav-z/free-image-generation-api`, где агент отправляет `POST {"prompt": "..."}` и получает лёгкую картинку;
- `OpenAI Images` - генерация через OpenAI Images API;
- `Custom / внешний генератор` - поле для заметок под будущий Replicate, Stability, ComfyUI или локальный workflow.

Практическая рекомендация: для публикаций использовать `OpenRouter Images` + `recraft/recraft-v3`, а Cloudflare Worker держать как бесплатный аварийный режим. Качество Cloudflare заметно ниже, особенно после сжатия размера ради стабильной работы с VPS.

### Рабочий Cloudflare Worker для картинок

Текущий worker:

```text

```

Проверка:


Ожидаемый ответ:


## Настройка площадок

Открыть:

```text
http://127.0.0.1:8088/admin/settings
```

### Telegram

Указать:

- токен бота;
- канал или несколько каналов для публикации;
- управляющий чат для команд бота.

Несколько каналов можно указать с новой строки или через запятую.

Если у черновика есть картинка, Telegram-публикация отправляется одним сообщением через `sendPhoto`: изображение и текст идут вместе, а текст становится подписью к картинке. Это важно для сценария, где бот Дзена забирает публикации из Telegram-канала: он видит один пост, а не отдельную картинку и отдельный текст.

Правила для такого режима:

- заголовок должен быть первой строкой текста;
- весь Telegram-текст должен помещаться в 1024 символа;
- если текст длиннее, агент не режет статью молча и не отправляет второе сообщение, а показывает ошибку и просит укоротить Telegram-версию.

### VK

Указать:

- `VK access token`;
- `owner_id`.

Для группы `owner_id` обычно начинается с минуса.

### VC и Дзен

Сейчас агент готовит материалы и сохраняет настройки для будущего автопостинга. Полную автоматизацию VC и Дзена лучше делать отдельным Playwright-сценарием с авторизованной браузерной сессией, потому что публичные API для публикации ограничены или нестабильны.

## ECC-практики для контент-агента

Из репозитория `affaan-m/ECC` точечно установлены Codex skills:

- `article-writing` - писать статьи от конкретного примера, факта или артефакта, без generic AI-вступлений.
- `brand-voice` - собирать и переиспользовать профиль авторского голоса из реальных текстов.
- `content-engine` - превращать одну сильную идею в платформенные форматы без потери голоса.
- `crosspost` - не отправлять одинаковый текст во все площадки, а адаптировать под ограничения Telegram, VK, VC, Дзена и блога.
- `eval-harness` - описывать критерии качества для новых агентных функций до реализации.
- `verification-loop` - прогонять build/test/lint/security/diff checks после существенных изменений.

Что забираем в логику проекта:

- Source-first: агент должен начинать от источника, артефакта, цифры, демо или личного наблюдения, а не от шаблонного вступления.
- Voice profile: стили в `/admin/styles` должны хранить не только “тон”, но и запреты, ритм, любимую структуру и стоп-слова автора.
- Crosspost без дублей: одна идея может уйти в несколько площадок, но текст должен отличаться по длине, хуку и структуре.
- Anti-slop gate: запрещены “game-changer”, “в современном быстро меняющемся мире”, служебные блоки, markdown-мусор, bait-вопросы и одинаковые концовки.
- Eval для агента: для новых кнопок вроде “Найти вирусные темы”, “Редакционный прогон”, “Опубликовать в выбранное” нужно фиксировать ожидаемый результат и регрессии.

Важно: весь ECC целиком не устанавливался, чтобы не получить дубли hooks, MCP и глобальных правил. Взяты только отдельные навыки, которые усиливают наш редакционный workflow.

## Cloud.ru Container Apps MCP

Документация `https://cloud.ru/docs/container-apps-evolution/ug/topics/guides__mcp-server` описывает MCP-сервер для управления Cloud.ru Evolution Container Apps и Artifact Registry через AI-агента.

Что умеет сценарий:

- подключить MCP-сервер `cloudru-containerapps-mcp` к агенту;
- создать Artifact Registry для Docker image;
- собрать и отправить Docker-образ приложения;
- создать Container Service или одноразовый Container Job;
- получить публичный адрес и логи контейнера;
- включить автодеплой новой ревизии при пуше нового image tag.

Установка MCP-сервера из инструкции:

```bash
go install github.com/Nick1994209/cloudru-containerapps-mcp/cmd/cloudru-containerapps-mcp@latest
```

Для работы нужны:

- доступ к Cloud.ru Foundation Models или совместимому агенту с MCP;
- сервисный аккаунт Cloud.ru;
- `CLOUDRU_KEY_ID`;
- `CLOUDRU_KEY_SECRET`;
- `CLOUDRU_PROJECT_ID`;
- Dockerfile приложения;
- Artifact Registry и Container Apps в одном проекте.

Для нашего проекта это полезно как альтернативный деплой-слой:

- staging-окружение для `AI на миллион`;
- одноразовые jobs для ежедневного поиска, SEO-аудита, IndexNow, проверки RSS;
- автодеплой контейнера после сборки нового Docker image;
- быстрый тест Cloud.ru вместо ручного VPS/systemd деплоя.

Ограничение: в документации Cloud.ru Container Apps указано, что собственный домен для обращения к контейнеру не используется, публичный адрес контейнера генерируется автоматически. Поэтому для основного домена `agent.gazon59.ru` пока логичнее оставить VPS/Cloudflare, а Cloud.ru рассматривать как staging/job-платформу или отдельный backend-сервис.

## Odysseus как ориентир для AI-лаборатории

Репозиторий `https://github.com/pewdiepie-archdaemon/odysseus` - свежий сильный сигнал рынка self-hosted AI workspace. По GitHub на 2026-06-02: создан 2026-05-31, около 29.7k stars, 3.6k forks, MIT, стек JavaScript + Python. README позиционирует проект как self-hosted версию опыта ChatGPT/Claude UI с локальными данными, агентами, памятью, deep research, сравнением моделей, документами, почтой, календарём и Cookbook для подбора локальных моделей.

Что полезно для `AI на миллион`:

- `Compare` - blind-test моделей для рерайта: один исходник, несколько моделей, скрытые варианты, ручной выбор лучшего результата.
- `Memory / Skills` - превратить наши правила стиля, удачные статьи и запреты в версионируемую память автора.
- `Deep Research` - усилить редакционный прогон: план исследования, источники, синтез, затем статья.
- `Cookbook` - страница подбора локальных моделей под железо Proxmox/VPS/MacBook: сколько VRAM/RAM, что запускать локально, что оставить через API.
- `AirLLM` - кандидат для экспериментального запуска больших моделей при малой VRAM. Не использовать как основной runtime без benchmark: экономит VRAM за счёт layer-wise loading, но может быть медленным и требовательным к диску.
- `Documents` - редактор, где AI не переписывает всё сам, а предлагает правки, версии и откат.
- `Notes & Tasks` - внутренние заметки/задачи агента: “проверить тему завтра”, “добрать источники”, “сделать пост в Telegram”.
- `Security Notes` - не выставлять такие AI-консоли напрямую в интернет; держать auth, reverse proxy, приватные порты и секреты отдельно.

Что не копировать целиком:

- Полный workspace с почтой, календарём и локальными моделями сейчас слишком широкий для нашего контент-агента.
- Нам нужен не универсальный ChatGPT-клон, а редакционная лаборатория: новости, рерайт, публикации, блог, рост канала.

Практический backlog:

1. Добавить `AI Compare` для черновика: OpenRouter/Gemini/HF/local модель генерируют варианты, пользователь голосует, результат сохраняется в историю. Реализовано как компактный блок прямо в черновике: кнопка `AI Compare`, карточки вариантов, применение победителя с откатом.
2. Добавить `Style Memory`: отдельная таблица удачных фраз, запретов, примеров и “не писать так”. Реализовано как `/admin/style-memory`, автоматически подмешивается к активному стилю.
3. Добавить `Research Report` для вирусной темы: источники, аргументы, риски, углы подачи, затем кнопка “сделать статью”. Реализовано в черновике кнопкой `Research Report`; отчёт сохраняется под материалом.
4. Добавить `Model Cookbook` для локального сервера: карточки моделей, требования, роль модели, команда запуска. Реализовано как `/admin/model-cookbook` с пресетами OpenRouter, Gemini, Hugging Face и Ollama/Qwen.
5. Добавить `Task Notes`: заметки агента по теме и отложенные follow-up действия. Реализовано в черновике и общей странице `/admin/task-notes`.

Идея статьи: “Почему Odysseus взлетел за два дня: люди хотят не ещё один чат с ИИ, а личную AI-операционку на своих данных”. Угол для нашего канала: связать хайп с практикой self-hosted AI, приватности, локальных моделей и агентных рабочих мест.

Принцип интерфейса: швейцарский нож, не кухонный комбайн. Новые возможности должны появляться как маленькие действия в существующем рабочем месте черновика или редакции, а не как отдельные тяжёлые комбайны. Каждый инструмент обязан иметь один понятный вход, один понятный результат и быстрый откат.

## Что стоит добавить следующим этапом

- Отдельные модели для разных задач: новости, рерайт, заголовки, фактчек, картинки.
- Blind-compare моделей для рерайта по примеру Odysseus Compare.
- Оценку тем по похожести на прошлые успешные публикации.
- Импорт старых постов автора из Telegram/VK/VC/Дзена.
- Автоматическую генерацию нескольких вариантов заголовков.
- Редакционный календарь по рубрикам.
- Playwright-публикацию в VC и Дзен через авторизованную браузерную сессию.
- Интеграцию с локальным ComfyUI или Replicate для картинок.
- Режим “утренний дайджест”: агент сам присылает в Telegram топ-5 тем и кнопки `черновик`, `пропустить`, `опубликовать`.
- Fact-check слой: отдельная проверка источников, дат, цифр и цитат перед публикацией.

## Короткая формула проекта

AI-редактор собирает инфоповоды, выбирает сильные темы, пишет черновики в стиле автора, помогает довести текст до публикации, добавляет картинку, публикует или ставит материал в очередь и сохраняет историю размещений.
